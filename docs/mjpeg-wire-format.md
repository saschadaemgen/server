# MJPEG wire format proof (S6-05)

Byte-for-byte documentation of what `GET /api/stream.mjpeg?src=<name>` emits.
Generated by `go run ./cmd/gen-docs` directly from the production
`internal/mjpeg.Writer` — if this drifts from the live server, the live
server has a bug, not this doc.

## HTTP response headers

```
HTTP/1.1 200 OK
Content-Type: multipart/x-mixed-replace; boundary=frame
Cache-Control: no-cache
Connection: close
Pragma: no-cache
Transfer-Encoding: chunked   (Go net/http default for streaming responses)
```

Critical: the Content-Type carries a NON-EMPTY `boundary=frame`
parameter. The ESP-Saison-4 bug was a server that shipped
`Content-Type: multipart/x-mixed-replace` without a boundary, causing
the ESP parser to see `Boundary: ''` and reject the stream. Our
endpoint cannot regress to that state — `TestHeaderContentType_BoundaryParseable`
feeds the header through `mime.ParseMediaType` and asserts the boundary
is parseable AND matches the body delimiter.

## Per-frame structure

Each frame on the wire:

```
--frame\r\n                       (boundary delimiter, leading dashes per RFC 2046 §5.1.1)
Content-Type: image/jpeg\r\n
Content-Length: <decimal>\r\n     (exact JPEG payload length, no padding)
\r\n                              (header/body separator)
<JPEG bytes>                      (Content-Length bytes verbatim)
\r\n                              (trailing CRLF before next boundary)
```

Notes for ESP parsers:

- All line terminators are `\r\n`. Bare `\n` does not occur outside the JPEG body.
- The `Content-Type: ` and `Content-Length: ` field names are followed by a SPACE, not just `:`.
- The boundary delimiter on the wire is `--frame`, i.e. `--` + the `boundary` parameter value.
- Frames are emitted back-to-back with no extra separator beyond the trailing `\r\n` of the previous frame; the next boundary `--frame\r\n` follows immediately.
- We do NOT emit a closing boundary (`--frame--`) — the connection ends when the client disconnects.

## Verification on a live server

```sh
curl -i -N 'http://localhost:8555/api/stream.mjpeg?src=mjpeg_bal' | head -c 600 | xxd
```

Should produce the same response headers + the same per-frame framing
shown above. The JPEG bytes will differ (live camera content), the
framing bytes will not.
